# 1. 문서 개요

## 1.1 문서명

AI + MCP 기반 건축설계 자동화 시스템 PRD

## 1.2 문서 목적

건축설계 프로세스를 AI 및 MCP 기반의 모듈형 시스템으로 재구성하기 위한 제품 요구사항을 정의한다. 대지 분석부터 기본설계까지의 설계 흐름을 하나의 자동화 파이프라인으로 연결하되, 각 단계를 독립적으로 실행 가능한 모듈로 분리하여 필요한 단계만 선택적으로 사용할 수 있도록 설계한다.

## 1.3 제품 비전

건축설계 업무를 단순 자동화하는 수준을 넘어, **설계자의 판단과 창의성을 보조하는 모듈형 설계 운영 시스템**을 구축한다.

- **전체 파이프라인 실행 방식**: 조사·분석부터 기본설계까지 순차적으로 연결하여 사용하는 방식
- **단계별 선택 실행 방식**: 특정 단계만 독립적으로 실행하거나, 필요한 단계만 조합하여 사용하는 방식

## 1.4 제품 범위

| 단계 | 명칭 | 역할 | 상태 |
| --- | --- | --- | --- |
| 1단계 | 조사·분석 | 대지 조건 및 법규 분석, 설계 파라미터 도출 | ⭐ MVP |
| 2단계 | 건축가능영역 생성 | 2D/3D 건축 가능 볼륨 생성 | ⭐ MVP |
| 3단계 | 기본구상 (컨셉) | 디자인 컨셉 및 형태 방향 설정 | 2차 |
| 4단계 | 계획설계 | 실제 건축 가능한 모델 구체화 | 3차 |
| 5단계 | 내부 실사화 | 내부 공간 분위기 및 재료감 시각화 | 3차 |
| 6단계 | 기본설계 | 도면 작성 가능 수준의 설계 완성 | 4차 |

## 1.5 문서 운영 원칙

이 PRD는 초기 완성 문서가 아니라, 실제 단계별 모듈을 구현해가면서 지속적으로 수정·보완되는 **살아있는 제품 문서**를 전제로 한다. 처음부터 모든 단계를 완벽하게 정의하지 않으며, 우선순위가 높은 단계부터 구현하면서 PRD를 업데이트한다.

# 2. 제품 목표

## 2.1 핵심 목표

- 건축설계 프로세스를 단계별 독립 모듈로 구성한다
- 전체 파이프라인 실행과 단계별 선택 실행을 모두 지원한다
- 각 단계의 결과를 재사용 가능한 구조화 데이터로 저장한다
- 단계별 프로토타이핑과 검증을 통해 PRD를 지속 개선할 수 있는 운영 구조를 만든다
- 설계 진행 현황, 입력 데이터, 산출물, 검토 상태를 한눈에 볼 수 있는 대시보드를 제공한다

## 2.2 기대 효과

- 설계 초기 조사 및 법규 검토 시간 단축
- 반복 계산 및 기준 검토 자동화
- 설계 대안 생성 속도 향상
- 단계별 산출물의 재사용성과 추적성 확보
- 실제 프로젝트 적용 과정에서 요구사항을 빠르게 고도화할 수 있는 운영 체계 확보

# 5. 제품 사용 방식

## 5.1 전체 파이프라인 실행

사용자는 대지 정보와 건축물 용도를 입력한 뒤, 1단계부터 6단계까지 순차적으로 실행할 수 있다. 각 단계의 산출물은 다음 단계의 입력으로 자동 연결된다.

## 5.2 단계별 선택 실행

| 상황 | 시작 단계 |
| --- | --- |
| 이미 법규 검토가 완료된 경우 | 2단계부터 시작 |
| 대략적인 볼륨이 있는 경우 | 3단계 컨셉부터 시작 |
| 계획설계 모델이 있는 경우 | 5단계 내부 실사화만 실행 |
| BIM 모델이 있는 경우 | 6단계 기본설계 정리만 실행 |

## 5.3 혼합 실행 예시

> 1단계 수동 검토 결과 업로드 → 2단계 자동 계산 → 3단계 디자이너 직접 컨셉 입력 → 4단계 이후 다시 자동화 흐름
>

# 6. 핵심 제품 요구사항

## 6.1 모듈 실행 구조

- 각 단계는 독립 실행 가능해야 한다
- 각 단계는 명확한 입력, 처리, 출력 구조를 가져야 한다
- 선행 데이터가 없으면 사용자가 직접 입력하거나 업로드할 수 있어야 한다
- 이전 단계 결과가 존재할 경우 자동으로 불러올 수 있어야 한다

## 6.2 단계 간 데이터 연결

- 각 단계의 산출물은 표준화된 구조(JSON)로 저장되어야 한다
- 다음 단계는 이전 단계 산출물을 그대로 사용하거나, 수정된 값을 반영해 사용할 수 있어야 한다
- 자동 생성 결과와 사용자 수정 결과를 구분해 관리해야 한다

## 6.3 PRD–제품 동기화 운영

- 각 단계 개발 시 실제 입력 형식, 예외 케이스, 출력 형식을 다시 문서에 반영해야 한다
- PRD에는 확정된 요구사항과 검토 중인 가설 요구사항을 구분해 기록할 수 있어야 한다
- 모듈 구현 후 발견된 문제는 제품 요구사항 업데이트 항목으로 관리해야 한다

## Rhino / Grasshopper 연동 방식 (결정 보류)

현재 Rhino/Grasshopper 연동 방식은 미확정 상태이다. MVP(1~2단계) 구현 후 결정한다. 현재 PRD에는 "Claude가 GH 스크립트 코드를 생성하고, 실행은 사용자가 수동으로 수행"하는 방식을 기본으로 문서화하되, 향후 아래 3가지 방식 중 하나로 확정한다.

| 방식 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| A. GH 스크립트 생성 (현재 기본값) | Claude가 .gh 파일 생성 → 사용자가 Rhino에서 수동 실행 | 추가 인프라 불필요, 즉시 구현 가능 | 자동화 루프 불완전 |
| B. RhinoCompute API | 서버에서 Rhino를 API로 호출해 자동 실행 | 완전 자동화 가능 | 라이선스 비용, 서버 설정 필요 |
| C. 로컬 Rhino MCP | 사용자 PC의 Rhino에 MCP가 명령 전달 | 로컬 자원 활용 | 사용자 환경 종속, 구현 복잡 |

# ⚙️ 기술 스택 & 인프라 (확정)

## 시스템 아키텍처 개요

```
Next.js (프론트엔드) ── HTTP/REST ──▶ FastAPI (백엔드 Python)
                                              │
                    ┌─────────────────────────┤
                    ▼                         ▼
        Vworld API / 법령정보센터 API     Claude API (분석·구조화)
        토지이음 크롤링
                    │
                    ▼
              SQLite (MVP) → PostgreSQL
                    │
                    ▼
    Next.js ◀── 단계 결과 JSON ── FastAPI
              (웹 화면에 리포트 표시)
```

## 기술 스택 상세

| 레이어 | 기술 | 역할 | 비고 |
| --- | --- | --- | --- |
| 프론트엔드 | Next.js (React) | 대시보드, 단계별 UI, 리포트 표시 | App Router, Tailwind CSS |
| 백엔드 | FastAPI (Python) | 단계 실행 API, MCP 호출, Claude API 호출 | 비동기(async) 구조 |
| DB (MVP) | SQLite + aiosqlite | 프로젝트 / 단계 결과 저장 | 파일 1개로 시작, 설치 불필요 |
| DB (이후) | PostgreSQL | 멀티프로젝트, 프로덕션 환경 | SQLAlchemy ORM으로 전환 용이 |
| AI | Claude API (Anthropic) | 법규 분석, 텍스트 구조화, GH 스크립트 생성 | claude-opus-4-6 모델 |
| 기하 계산 | Shapely + GeoPandas | 대지 경계 처리, Setback 계산 | Python 백엔드에서 실행 |
| 이미지 생성 | DALL-E 3 (OpenAI API) | 3·5단계 컨셉/인테리어 이미지 | MVP 이후 적용 |
| 크롤링 | BeautifulSoup / httpx | 토지이음 페이지 파싱 (API 키 발급 전) | API 키 발급 후 대체 |

| error_message | TEXT | 오류 메시지 |
| created_at | DATETIME | 생성일시 |
| updated_at | DATETIME | 수정일시 |

## 1단계 리포트 출력 형태 (웹 화면 표시)

1단계 산출물은 **웹 대시보드에 직접 렌더링**된다. 설계자는 별도 다운로드 없이 화면에서 바로 검토·수정 가능해야 한다.

- **① 대지 개요** — 주소, 지목, 면적, 용도지역·지구 (Vworld 데이터 기반)
- **② 설계 파라미터 카드** — 건폐율, 용적률, 높이제한, 이격거리, 조경, 주차 (각 수치 + 법적 근거 링크)
- **③ 법규 요약** — 인허가 / 구조 / 피난 / 친환경 기준별 요약 텍스트 (Claude 분석)
- **④ AI 신뢰도 경고** — 자동 수집 항목과 검토 필요 항목을 색상으로 구분 표시
- **⑤ 수정 인터페이스** — 각 수치를 인라인 편집 가능 (수동 수정 시 "manual" 태그 자동 부여)

# 7. 단계별 요구사항

## Stage 1 — 조사·분석 ⭐ MVP

대지에 적용되는 건축 관련 법규 및 대지 조건을 조사·분석하여 설계에 필요한 법적 요구사항과 설계 파라미터를 정리한다.

**입력**

| 구분 | 항목 |
| --- | --- |
| 필수 | 건축 예정 주소, 건축물 용도 |
| 선택 | 참고 문서 / PDF 자료, 관련 웹 링크, 수동 입력한 법규 검토 결과 |

**산출물**: 대지 분석 데이터 (JSON), 법규 정리 데이터 (JSON), 설계 법적 요구사항, 설계 파라미터 (JSON)

**처리 요구사항**

- 대지 위치 및 기본 정보를 식별해야 한다
- 용도지역, 용도지구, 용도구역을 정리해야 한다
- 접도조건, 지목, 규제사항을 분석해야 한다
- 적용 가능한 법률 및 조례를 식별해야 한다
- 법적 요구사항을 인허가, 구조, 피난, 접근성, 친환경 기준으로 구분해야 한다
- 2단계에서 사용할 설계 파라미터를 구조화해야 한다

**🔧 1단계 도구 구조**

| 분류 | 도구 | 설명 |
| --- | --- | --- |
| MCP | Vworld Open API | 주소 입력 → 지번 좌표, 필지 정보(지목·면적), 용도지역 조회. 국토부 공공데이터 기반의 무료 API |

| Python | 파라미터 검증 / 계산 | 여러 출처의 수치를 교차 검증. 건폐율·용적률 산식 확인, 이격거리 계산 등 수치 기반 처리 |
| 출력 포맷 | JSON (설계 파라미터 스키마) | `{ "buildingCoverage": 60, "floorAreaRatio": 200, "maxHeight": 20, "setbacks": {"front":3,"side":1.5,"rear":1.5}, "parkingRequired": 1, "landscapeRatio": 10 }` |

**후속 연결 데이터**: 건폐율 · 용적률 · 높이 제한 · 이격거리 · 사선 제한 · 조경 면적 · 주차 대수 · 기타 대지 규제조건

## Stage 2 — 건축가능영역 생성 ⭐ MVP

1단계의 설계 파라미터를 적용하여 대지 위에 건축 가능한 영역을 2D 및 3D 형태로 생성한다.

**입력**

| 대체 입력 (독립 실행 시) | 주소 (→ Vworld API로 필지 경계 자동 조회), DXF / GeoJSON 파일 직접 업로드, 건폐율 / 용적률 / Setback 수동 입력 |

**처리 요구사항**

- 대지 경계를 계산 가능한 형상 데이터로 정리해야 한다
- 건폐율, 이격거리, 공지, 조경, 주차 조건을 반영한 2D 건축가능영역을 생성해야 한다
- 높이 제한과 사선 제한을 반영해야 한다
- 용적률을 고려하여 최대 건축가능 볼륨을 생성해야 한다

**🔧 2단계 도구 구조**

| 분류 | 도구 | 설명 |

| Claude | 결과 해석 + 사용자 설명 | 생성된 건축가능영역과 볼륨을 설계자에게 자연어로 설명. 주요 제약 조건과 설계 주의사항을 정리해 제공 |

**후속 연결 데이터**: 건축가능영역 (GeoJSON) · 최대 건축가능 볼륨 · 초기 매스 3D 데이터

# 8. 대시보드 요구사항

## 8.1 대시보드 목적

프로젝트별 설계 자동화 진행 상태를 관리하고, 각 단계의 입력·출력·검토 상태를 한눈에 파악할 수 있는 운영 화면이어야 한다.

## 8.2 핵심 기능

- 프로젝트 목록 조회 및 단계 진행 현황 표시
- 단계별 실행 가능 여부 및 입력값 상태 표시
- 단계별 산출물 조회 및 검토 상태 표시
- 단계별 재실행 및 수정 진입
- 수동 입력 데이터와 자동 생성 데이터 구분 표시

## 8.3 단계 상태 유형

`미시작` / `입력대기` / `실행완료` / `검토중` / `수정필요` / `확정`

## 8.4 권장 화면 구성

| 영역 | 내용 |

# 11. 제품 개발 및 문서 운영 방식

## 11.1 우선 개발 전략 (확정)

**MVP 범위: 1단계 + 2단계**. 코어 파이프라인(조사·분석 → 건축가능영역 생성)을 먼저 완성하고 실제 프로젝트에서 검증한 뒤 나머지 단계를 순차 확장한다.

1. ⭐ MVP — 1단계 조사·분석 (Vworld API + 법규 크롤링 + 파라미터 추출)
2. ⭐ MVP — 2단계 건축가능영역 생성 (Shapely 기하 계산 + 다이어그램)
3. 대시보드 1차 버전 (1~2단계 상태 관리)
4. 3단계 컨셉 (DALL-E 이미지 생성 연동)
5. 4~6단계 확장 (GH 스크립트 생성 + 공사비 산출)

## 11.2 단계별 PRD 업데이트 항목

- 실제 입력 포맷 / 예외 상황 / 사용자 수정 포인트
- 출력 데이터 구조 / 후속 단계 연결 방식
- 도구 실제 작동 여부 및 대안 도구 검토 결과
- 추가 요구사항

## 11.3 권장 문서 구조

| 문서 유형 | 역할 |
| --- | --- |
| Master PRD | 전체 시스템 구조, 공통 원칙, 도구 아키텍처 관리 (이 문서) |
| Stage PRD | 각 단계별 상세 요구사항, 실제 입/출력 스키마 문서 |
| Change Log | 구현 중 변경된 요구사항과 결정사항 기록 |
| Dashboard PRD | 운영 화면 및 상태 관리 기능 정의 |
